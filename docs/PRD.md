# PRD: Personal Time Management Software MVP

## 1. Product Overview

This product is a personal time management tool for tracking what needs to be done, how much time each goal is expected to take, and how much time has actually been invested each day.

The product connects two core views:

- A Calendar page for planning the next day and recording actual daily time entries.
- A Project Management page for managing goals, projects, estimated effort, and progress.

The core idea is to make long-term goals measurable through accumulated time investment. For example, the user can create a 60-hour goal, ask AI to break it into smaller time-boxed tasks, place those tasks into a planned schedule through an ICS file, and then complete the goal by logging actual focused hours in the Calendar.

## 2. Product Goals

### Core Goal

Help the user turn abstract personal goals into trackable time-based progress.

### MVP Goals

The first version should allow the user to:

- Create projects or goals with estimated total hours.
- Ask AI to break a goal into smaller tasks with estimated durations.
- Add planned task blocks to a schedule, using ICS as an intermediate format.
- Record what they did each day and how long it took.
- Use a built-in Pomodoro timer to create accurate actual work records.
- Link daily time records to specific projects.
- Automatically calculate invested time, remaining time, and completion progress.
- View planned time and actual time from both the Calendar perspective and the Project perspective.

## 3. Target User

### Primary User

The primary user is an individual who wants to manage personal goals, projects, study plans, habits, or long-term work through time investment.

### User Characteristics

- Wants to know where daily time is spent.
- Has long-term goals but lacks a clear sense of progress.
- Wants more than a simple to-do list.
- Wants to complete large goals through consistent daily effort.

## 4. User Pain Points

### Abstract Goals Lack Progress Feedback

Many goals exist as vague intentions, such as learning a skill, finishing a course, writing, preparing for an exam, or building a personal project. Without a measurable unit, it is hard to know how far the user has progressed.

### Calendar and Task Management Are Usually Separate

Calendar tools usually track time blocks. Task tools usually track pending work. The two are often disconnected, so time spent on a task does not automatically update the larger project or goal. They also rarely distinguish clearly between what was planned and what was actually done.

### Long-Term Effort Is Hard to Quantify

For study, writing, research, fitness, portfolio building, and similar goals, progress often comes from repeated time investment rather than one-time task completion. The user needs to see accumulated hours and remaining effort.

### Goals Are Hard to Break Down Manually

After creating a large goal, the user may not know how to divide it into concrete tasks or how much time each task should take. AI can help generate an initial breakdown that the user can edit before placing tasks into a planned schedule.

## 5. Core Product Model

The product is built around four core objects.

### Project / Goal

A Project or Goal represents a long-term thing the user wants to complete.

Examples:

- Learn Python
- Write a novel
- Prepare for an exam
- Build a personal website
- Complete a 100-hour fitness plan

Project fields:

| Field | Description |
| --- | --- |
| Name | Project or goal name |
| Description | Optional details |
| Estimated total hours | Total expected time investment |
| Invested hours | Automatically calculated from linked time entries |
| Remaining hours | Estimated total hours minus invested hours |
| Progress | Invested hours divided by estimated total hours |
| Status | Not started, in progress, completed, or paused |
| AI-generated task plan | Optional generated task breakdown |

### Task

A Task is an optional breakdown item under a project.

Example:

Project: Learn Python

- Watch basic syntax lessons, estimated 10 hours
- Complete practice exercises, estimated 15 hours
- Build a small project, estimated 20 hours

Task fields:

| Field | Description |
| --- | --- |
| Name | Task name |
| Project | Parent project |
| Estimated hours | Expected task effort |
| Invested hours | Automatically calculated from linked time entries |
| Status | Not started, in progress, completed, or paused |

For MVP, basic task-level planning should be supported when tasks are generated by AI. Full manual task management can be expanded later.

### Planned Time Block

A Planned Time Block represents what the user intends to do on a specific day. It belongs to the planning schedule, not the actual execution record.

Examples:

- 2026-05-26, 09:00-11:00, Learn Python, Watch basic syntax lessons
- 2026-05-26, 15:00-16:00, Writing, Outline chapter 1

Planned time block fields:

| Field | Required | Description |
| --- | --- | --- |
| Date | Yes | Scheduled date |
| Start time | Yes | Planned start time |
| End time | Yes | Planned end time |
| Duration | Auto | Calculated from start and end time |
| Project | Yes | Linked project |
| Task | No | Optional linked task |
| Source | No | Manual, AI-generated, or imported from ICS |
| Notes | No | Optional plan details |

Planned Time Blocks do not count toward project invested hours.

### Time Entry

A Time Entry records what the user actually did on a given day. Only Time Entries count toward project invested hours and completion progress.

Examples:

- 2026-05-25, Learn Python, 2 hours
- 2026-05-25, Writing, 1.5 hours

Time entry fields:

| Field | Required | Description |
| --- | --- | --- |
| Date | Yes | Defaults to the selected day or today |
| Project | Yes | Linked project |
| Task | No | Optional linked task |
| Duration | Yes | Time spent, such as 2 hours |
| Start time | No | Optional start time |
| End time | No | Optional end time |
| Source | No | Manual, Pomodoro timer, or converted from planned block |
| Notes | No | What was done during the session |

## 6. Functional Requirements

## 6.1 Calendar Page

The Calendar page is used to plan intended time usage and record actual time investment.

The page should show two separate columns for each selected day:

- Plan: the schedule prepared in advance, usually for the next day.
- Actual: the execution record entered during or after the day.

The two columns can use the same visual structure, but they represent different data. Planned items are intentions. Actual items are completed execution records.

### Required Capabilities

- View time entries by day.
- Add, edit, and delete planned time blocks.
- Add, edit, and delete actual time entries.
- Link each planned block or actual entry to a project.
- Enter duration manually for actual entries.
- Start a Pomodoro timer from the Actual column.
- Automatically create an actual time entry when a Pomodoro session is completed.
- Add optional notes.
- Show total planned time for the selected day.
- Show total actual time for the selected day.
- Show the time distribution across projects for both plan and actual columns.
- Allow the user to convert or copy a planned block into an actual time entry.
- Ensure only actual time entries update project progress.

### Time Entry Creation Fields

| Field | Required | Notes |
| --- | --- | --- |
| Date | Yes | Defaults to selected date |
| Project | Yes | Selected from existing projects |
| Task | No | Can be omitted in MVP |
| Duration | Yes | Example: 2 hours |
| Notes | No | Description of what was done |

### Calendar Day Example

```text
2026-05-25

Plan                              Actual
Total planned: 5 hours            Total actual: 4.5 hours

Learn Python        2 hours       Learn Python        2 hours
Writing Project     2 hours       Writing Project     1.5 hours
Fitness             1 hour        Fitness             1 hour
```

At the end of the day, only the Actual column is used to calculate invested hours and project progress.

## 6.2 Project Management Page

The Project Management page is used to manage goals, projects, and planned effort.

### Required Capabilities

- Create a project.
- Edit a project.
- Delete or archive a project.
- Set estimated total hours.
- View invested hours.
- View remaining hours.
- View completion percentage.
- View all time entries linked to a project.
- View all planned blocks linked to a project.
- Update project status.
- Ask AI to generate a task breakdown from the project goal and estimated total hours.
- Review, edit, accept, or reject AI-generated tasks.
- Export accepted planned tasks as an ICS file.
- Import an ICS file to create planned time blocks.

### Project Fields

| Field | Required | Notes |
| --- | --- | --- |
| Project name | Yes | Example: Learn Python |
| Estimated total hours | Yes | Example: 60 hours |
| Description | No | Additional context |
| Status | Yes | Defaults to in progress |
| Created at | Auto | Generated by system |
| Invested hours | Auto | Calculated from Calendar entries |
| Remaining hours | Auto | Estimated total hours minus invested hours |
| Progress | Auto | Invested hours divided by estimated total hours |

### Project Detail Example

```text
Learn Python

Estimated total: 60 hours
Invested: 12 hours
Remaining: 48 hours
Progress: 20%

Recent entries:
2026-05-25  2 hours    Watched basic syntax lessons
2026-05-24  1.5 hours  Practice exercises
```

## 6.3 AI Task Breakdown

After creating a project or goal, the user can ask AI to break the goal into smaller tasks with concrete estimated durations.

### Required Capabilities

- Generate a list of tasks from project name, description, estimated total hours, and optional user instructions.
- Assign an estimated duration to each generated task.
- Ensure the sum of generated task estimates roughly matches the project estimated total hours.
- Let the user edit generated task names, durations, and notes before accepting them.
- Let the user regenerate the breakdown if the result is not useful.
- Save accepted tasks under the project.

### AI Task Breakdown Example

```text
Project: Learn Python
Estimated total: 60 hours

Generated tasks:
1. Python syntax basics          8 hours
2. Data structures               10 hours
3. Functions and modules         8 hours
4. File handling                 6 hours
5. Practice exercises            12 hours
6. Build a small project         16 hours
```

## 6.4 ICS-Based Planning

The product should use ICS files as an intermediate format for planned schedule data.

### Required Capabilities

- Export selected planned tasks or planned time blocks to an ICS file.
- Import an ICS file and convert events into planned time blocks.
- Preserve event title, date, start time, end time, duration, and notes where available.
- Attempt to map imported ICS events to existing projects or tasks.
- Let the user confirm mappings before saving imported planned blocks.

### ICS Planning Rules

- ICS data creates Planned Time Blocks by default.
- ICS-imported items must not create Actual Time Entries automatically.
- ICS-imported items must not update project invested hours.
- If a planned block is completed, the user must explicitly confirm or convert it into an actual time entry.

## 6.5 Pomodoro Timer

The product should include a built-in Pomodoro timer for creating more accurate Actual time entries.

The timer belongs to the execution side of the product. It records work that actually happened, not work that was merely planned.

### Required Capabilities

- Start a Pomodoro session from the Actual column.
- Select a project before starting the timer.
- Optionally select a task before starting the timer.
- Use a default focus duration, such as 25 minutes.
- Allow the user to configure focus duration.
- Allow the user to pause, resume, or stop a session.
- When a session is completed, automatically create an Actual time entry.
- Mark the time entry source as Pomodoro timer.
- Add optional notes before or after the session is saved.

### Pomodoro Recording Rules

- Completed focus sessions create Actual Time Entries.
- Actual Time Entries created by Pomodoro sessions count toward project invested hours.
- Break time should not count toward project invested hours.
- If a session is stopped early, the user should choose whether to save the elapsed focus time or discard it.
- If a session is paused, paused time should not count toward the recorded duration.

### Pomodoro Example

```text
Project: Learn Python
Task: Python syntax basics
Timer: 25 minutes

After completion:
2026-05-25
Learn Python
Python syntax basics
0.42 hours
Source: Pomodoro timer
```

## 6.6 Calendar and Project Sync

This is the core product behavior.

When the user creates an Actual time entry and links it to a project, the project should automatically update:

- Invested hours increase.
- Remaining hours decrease.
- Completion progress updates.
- The time entry appears in the project detail history.

If an Actual time entry is edited or deleted, all related project statistics must be recalculated.

When the user creates, edits, deletes, imports, or exports a Planned time block, project invested hours should not change.

When a Pomodoro timer session creates an Actual time entry, the linked project should update in the same way as a manually created Actual time entry.

### Sync Example

Initial project:

```text
Project: Learn Python
Estimated total: 60 hours
Invested: 0 hours
```

Calendar entry:

```text
2026-05-25
Learn Python
2 hours
```

Updated project:

```text
Learn Python
Estimated total: 60 hours
Invested: 2 hours
Remaining: 58 hours
Progress: 3.3%
```

## 7. MVP Scope

### Must Have

- Create, edit, and delete projects.
- Set estimated total hours for each project.
- Ask AI to generate task breakdowns with estimated durations.
- Review and edit AI-generated tasks before saving.
- Create, edit, and delete planned Calendar blocks.
- Create, edit, and delete actual Calendar time entries.
- Create actual time entries through a built-in Pomodoro timer.
- Link planned blocks and actual time entries to projects.
- Automatically calculate project progress.
- Calendar day view with separate Plan and Actual columns.
- Use ICS as an import/export format for planned schedule blocks.
- Project list page.
- Project detail page.

### Out of Scope for MVP

- Team collaboration.
- Multi-user accounts.
- Complex permissions.
- Third-party calendar sync.
- Report export.
- Native mobile app.

## 8. Core User Flows

### Flow 1: Create a Goal

1. User opens the Project Management page.
2. User clicks New Project.
3. User enters project name: Learn Python.
4. User enters estimated total hours: 60.
5. User saves the project.
6. System creates the project and shows its progress page.

### Flow 2: Generate Tasks with AI

1. User opens a project detail page.
2. User clicks Ask AI to Break Down Tasks.
3. System uses the project name, description, and estimated total hours to generate task suggestions.
4. User reviews generated task names and durations.
5. User edits any task if needed.
6. User accepts the task breakdown.
7. System saves the tasks under the project.

### Flow 3: Add Tasks to Planned Schedule Through ICS

1. User selects generated tasks or planned blocks.
2. User chooses Export to ICS.
3. System creates an ICS file containing planned events.
4. User imports the ICS file into the planning schedule or another calendar tool.
5. If imported back into the product, system creates Planned Time Blocks.
6. Planned Time Blocks appear in the Plan column and do not affect project progress.

### Flow 4: Record Daily Effort

1. User opens the Calendar page.
2. User selects the current date.
3. User reviews the Plan column.
4. User creates or converts an item into an Actual time entry.
5. User selects project: Learn Python.
6. User enters duration: 2 hours.
7. User enters notes: Watched basic syntax lessons.
8. User saves the entry.
9. System updates the Actual column and linked project progress.

### Flow 5: Record Work with Pomodoro Timer

1. User opens the Calendar page.
2. User chooses the Actual column.
3. User starts a Pomodoro session.
4. User selects project: Learn Python.
5. User optionally selects task: Python syntax basics.
6. User works until the timer completes.
7. System automatically creates an Actual time entry using the completed focus duration.
8. System updates the linked project progress.

### Flow 6: Review Goal Progress

1. User opens the Project Management page.
2. User selects Learn Python.
3. User views invested time, remaining time, and progress percentage.
4. User reviews historical time entries.
5. User decides whether to continue, pause, or adjust the goal.

## 9. Non-Functional Requirements

### Usability

- Adding a time entry should be fast.
- A time entry should require no more than three required fields: date, project, and duration.
- Project progress should be visible at a glance.
- Navigation between Calendar and Project Management should be simple and predictable.

### Data Accuracy

- Project invested hours must be calculated from linked time entries.
- Editing or deleting a time entry must update project statistics.
- Changing estimated total hours must recalculate completion percentage.
- Duration should support decimal hours, such as 1.5 hours.
- Planned time and actual time must be stored separately.
- Planned time must never be counted as invested time unless the user explicitly converts or records it as actual time.
- ICS import must create planned blocks by default, not actual time entries.
- Pomodoro break time and paused time must not be counted as actual invested time.

### AI Reliability

- AI-generated task plans must be editable before saving.
- The product should show generated task durations clearly.
- The user must confirm AI-generated tasks before they affect the project plan.

### Local-First

Because this is initially a personal tool, the MVP can prioritize local storage before cloud sync or account systems.

## 10. Metrics

- Daily total invested time.
- Daily planned time.
- Daily Pomodoro focus time.
- Weekly total invested time.
- Weekly planned time.
- Weekly Pomodoro focus time.

## 13. Current Prototype Addendum

This section documents product decisions and prototype behavior added after the initial MVP PRD.

### 13.1 Local Desktop-Style App

The prototype should be opened through a local Node.js service rather than a `file://` URL.

Required behavior:

- The app runs at `http://localhost:5173`.
- `start-local.cmd` starts the local service.
- `open-app.cmd` opens the app through a desktop-style launcher.
- AI features and local state sync depend on the local service being active.
- Users should not rely on opening `index.html` directly through `file://` for normal use.

### 13.2 Local State Sync

The prototype should store user data in a local JSON state file so the app and local assistant can inspect and update the same data.

Required behavior:

- Browser state synchronizes to `data/state.json`.
- The state file includes projects, tasks, planned blocks, actual entries, and daily reviews.
- On first migration, existing browser state should not be overwritten by an empty server state.
- Local assistant workflows can read and write the state file when the user asks for direct project/calendar edits.

### 13.3 Task-Level Calendar References

Planned blocks and actual entries should reference a project family and, where applicable, a specific task under that project.

Required behavior:

- Calendar planning selects a project first, then a task under that project.
- Planned blocks should store both `projectId` and `taskId`.
- Actual entries should store `projectId` and may store `taskId`.
- Project progress is still calculated from actual entries linked to the parent project.
- Week/month calendar blocks should visually retain their project family through color.
- Project cards, planned entries, actual entries, and calendar blocks should use consistent project-family color cues.

### 13.4 Week and Month Calendar Views

The calendar should focus on week and month planning rather than a standalone day tab.

Required behavior:

- Calendar supports week and month view modes.
- Selecting a date in week/month view updates the selected-day plan/actual editor.
- Clicking a planned block in week/month view selects its date and asks whether to delete the block.
- Planned time and actual time totals are visible for each calendar date.

### 13.5 AI Task Breakdown Conversation

AI task breakdown should support iterative refinement instead of one-shot generation only.

Required behavior:

- User can generate an initial task breakdown from project name, description, and estimated total hours.
- User can send follow-up instructions such as "make the tasks smaller" or "limit each block to 1.5 hours."
- AI returns an editable task draft.
- User can edit task name, estimated hours, and notes after generation.
- Saved tasks remain editable with explicit Edit/Done controls.
- AI responses in the Chinese prototype should use Simplified Chinese.

### 13.6 AI-Assisted Scheduling

Accepted or drafted project tasks can be scheduled into future planned calendar blocks.

Required behavior:

- AI scheduling reads current tasks and existing planned blocks.
- Generated planned blocks should reference specific task IDs where possible.
- Long tasks should be split into smaller blocks, preferably no more than 2 hours unless instructed otherwise.
- AI scheduling creates planned blocks only and never creates actual time entries automatically.
- If AI is unavailable, the app should fall back to a deterministic local scheduler.

### 13.7 Pomodoro Focus Mode

The Pomodoro timer should provide a full-screen focus experience.

Required behavior:

- Starting a Pomodoro opens a full-screen focus overlay.
- The focus overlay shows a large countdown timer, circular progress indicator, linear progress indicator, and selected project/task context.
- The focus overlay includes a notes area for writing what the user is doing during the session.
- Stopping and saving the timer creates an actual time entry with elapsed focus duration and notes.
- The user can pause/resume, stop and save, discard, or collapse the full-screen overlay while keeping the timer running.

### 13.8 Daily Review

The app should support an AI-assisted daily review after the user has completed or updated a day's actual entries.

Required behavior:

- Daily Review reads today's planned blocks, today's actual entries, past 7 days of actual entries, future 7 days of planned blocks, and project progress.
- The app calculates planned hours, actual hours, and completion rate.
- AI returns a concise review containing completion rate, wins, gaps, risks, draft adjustment suggestions, and next focus.
- AI adjustment suggestions are drafts only; the app should not automatically modify future plans unless the user explicitly applies changes.
- If AI is unavailable, the app should still show a local fallback review.

### 13.9 ICS Behavior

ICS remains an intermediate format for planned schedule data.

Additional behavior:

- ICS-imported events create planned blocks by default.
- Imported events should attempt to map to existing project families and tasks where possible.
- ICS import must not create actual entries or update invested time.
